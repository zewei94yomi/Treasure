# 开发复盘：25 轮迭代的经验、心得与不足

> 2026-06 ~ 2026-07，人机结对开发（设计/验收：Zewei；实现：Claude）。本文是封存前的诚实复盘。

## 一、数字面板

- **25 轮**功能迭代，**32 次提交**，每轮一个原子 commit（天然的回滚粒度）
- **~11,300 行 JS**（19 个文件）+ 826 行 CSS + 488 行 HTML，**零依赖、零构建**
- **237 条回归断言**，每轮结束全绿才发版
- **4 种玩法**、21 种怪 + 3 Boss（各带技能）、12 位可招募英雄、20 个割草技能、16 把武器、72 件宝物、**4 层调参体系约 250 根滑杆**
- 资产全部 **CC0**：Kenney（皮肤/准星）、Dungeon Crawl（怪物贴图）、qubodup/bart（音效）、Google Fonts 自托管子集

## 二、做对了什么（值得复用的方法论）

**1. 零依赖 + 静态部署 = 迭代速度的地板极高。**
没有 npm、没有构建、没有环境漂移。改一行→刷新→看到。25 轮里没有一分钟花在"工程链路坏了"上。对单人/结对的小型项目，这个选型的复利巨大。

**2. 数据驱动是内容型游戏的命脉。**
武器/怪物/升级卡/英雄全部是数据表+少量行为旗标。第 21 轮加"手雷技能全家桶"只花了一张卡定义+一段引擎；第 25 轮"旺财入池"是一行数据。**内容的边际成本必须趋近于零，否则内容型游戏做不动。**

**3. 调参面板是单人开发者的最高杠杆功能。**
"玩家即策划"工作流：游戏内滑杆实时改数值 → 存档持久化 → 玩到满意 → 一条指令把覆盖值固化为代码默认。全程不需要开发者在场，平衡收敛速度是改代码重发的十倍。四层面板（全局/英雄/怪物/技能）+ 练习场学习台，最终把整个游戏变成了自己的编辑器。

**4. 回归断言是跨轮开发的安全网。**
237 条"当前态快照"断言（数据完整性、代码接线字符串、版本一致性）。它们简陋，但每一轮都真实拦下过跨轮破坏——尤其是"改了数量忘了改引用"这类。**快照式断言的过期报警本身就是价值**：它逼你确认每个改动的影响面。

**5. 浏览器泵帧 E2E。**
后台标签 rAF 冻结曾是 E2E 的死路，`for(){ t+=16.7; g.frame(t) }` 手动泵帧解决之。此后每轮都能做"快进 10 秒验证复活倒计时/DDA 收敛/天气轮换"这类时间性验证，报告里的"实测 566→83"全部来自它。

**6. 截图驱动的需求与验收。**
用户每轮用截图+红框标注提需求，比文字精确一个数量级（"经验条放到这里"的红框箭头直接决定了坐标）。反向地，实现方交付时也回截图证明。**视觉产品用视觉沟通。**

**7. 修一类问题，而不是一个问题。**
"站在枪上来回横跳"修成 needExit 状态机；"高亮不一致"统一四个面板的行为；"锚点错插"之后所有补丁锚点强制带上下文。每次事故沉淀为惯例，后期轮次的返工率显著下降。

## 三、心得（认知层面的收获）

**算法换代 > 参数修补。** 怪物卡地形修了三轮参数（分离/侧滑/强制寻路），每次"好一点"；第 22 轮换成多源 BFS 流场，一次根治且更省性能。当同一问题第三次出现，该怀疑的是方法而不是数值。

**手感 = 延迟 + 反馈。** 准星从画布搬到 DOM（零帧延迟）、命中音效加"肉"、hit-pause、击退、挥空提示——这些不改变任何数值的改动，对"爽"的贡献超过大多数平衡调整。

**共享可变状态是重构之敌。** "双人升级分离"是全项目最贵的一次改动：hordeState 单例拆成 per-player 触碰 60+ 处引用。反例是对决模式：因为从第一行就按 owner 建模，几乎没有返工。**归属（ownership）要在系统诞生时就建模，而不是需要时再拆。**

**"不存在"也是 bug。** 旺财三轮不见踪影，最后发现是它的招募卡从未进过卡池——玩家视角的"它坏了"和代码视角的"没这回事"之间隔着一次认真的链路核查。Debug 请求要先证明功能路径存在，再谈修。

**设计稿的真相在 inline style 里。** UI 重做两轮的效率来自设计交接包"每个元素的内联样式即精确规格"——像素级照抄远快于凭感觉还原。

**动画的结束态必须等于静止态。** 升级卡闪缩、血条不同步，两次视觉 bug 同根：过渡/动画与最终布局各说各话。

## 四、不足（诚实清单）

1. **game.js 成了 4300 行上帝对象。** 战斗/渲染/HUD/对决/寻路全在一个类里。每轮"再加一点"的路径依赖所致。若继续开发，第一件事应是按系统拆模块（哪怕仍是全局脚本，先拆文件）。
2. **测试是字符串快照，不是行为验证。** `includes("公式片段")` 能抓"忘了改"，抓不住"改错了"。缺一层无头行为测试（模拟 N 帧断言状态），泵帧法其实已经铺好了路，但从未沉淀进 CI。
3. **平衡靠体感，没有遥测。** 没有记录任何对局数据（存活曲线、技能选取率、死因分布），DDA 的 0.6 血线基准是拍脑袋值。哪怕一个本地 CSV 导出都会让平衡讨论有据可依。
4. **移动端从未启动。** 俯视双摇杆玩法天然适触屏，但键盘中心的输入层没有留抽象缝隙，触屏是个中等规模改造。
5. **美术三源混杂。** emoji + CC0 贴图 + 程序绘制三种保真度并存，整体气质靠 UI 风格勉强缝合。资产管线（统一调色/描边/尺寸）应该更早建立。
6. **没有 LICENSE 文件。** 资产是 CC0，但仓库代码本身未声明许可。封存前建议补一个（MIT 即可），否则法律上默认"保留所有权利"。
7. **存档无版本迁移机制。** 键名里的 v1 从未演进，字段全靠"读不到就用默认"兜底。真出现破坏性结构变更会很难受。
8. **性能没有压测过上限。** hordeCap=80 下 60fps 稳定，但 200 上限滑杆只是"能拉"，没验证过低端机表现。
9. **对决模式浅。** 本地同键盘双人已尽力（空投/回合制），但没有手柄支持，键盘鬼影是物理硬伤。

## 五、如果重来一次

1. **第 1 天就建调参面板**（哪怕只有 5 根滑杆）——它后来改变了整个协作方式，越早越赚。
2. **模块边界先行**：Game 只做编排，战斗/渲染/HUD/模式各自成文件，事件总线代替直接引用。
3. **行为测试与泵帧法进 CI**，字符串断言只做数据完整性。
4. **输入抽象层**（动作→按键映射已有，再加"动作→触摸/手柄"的插槽）。
5. **每轮结束写复盘小节**（本项目从第 17 轮才开始在 memory 里沉淀"坑"，前 16 轮的教训靠重新踩）。

## 六、结语

这个项目最大的产出不是游戏本身，而是**一套被验证过的人机结对开发工作流**：截图提需求 → 任务清单 → 数据驱动实现 → 泵帧实测 → 快照断言 → 原子提交 → 线上验证 → 逐项中文报告。25 轮零烂尾，每一轮结束时线上都是可玩的。

游戏封存于 v31，GitHub Pages 继续在线。Quack. 🦆
